Optimizing Redis cache invalidation strategies for high-throughput WebSocket events

Asked 3 days ago Active today Viewed 12k times
42

We are currently seeing a significant latency spike during high-throughput WebSocket events where cache invalidation is triggered across multiple clusters. The current strategy relies on a simple TTL-based expiration, but this leads to "thundering herd" problems when keys expire simultaneously.

Has anyone successfully implemented a probabilistic cache invalidation or a more robust pub/sub model to handle 100k+ concurrent connections without saturating the Redis I/O?

// Current implementation causing bottlenecks
async function invalidate(key) {
  const result = await redis.del(key);
  if (result) {
    logger.log(`Invalidated: ${key}`);
    broadcastEvent('cache_flush', { key });
  }
}

You should look into the XADD command in Redis Streams. Unlike the standard Pub/Sub model which is fire-and-forget, Streams provide a persistent log that allows consumers to catch up if they miss an invalidation window.

For the thundering herd, implement a Jitter-based TTL. Instead of a fixed 300s, use 300 + rand(0, 30). Here is a cleaner implementation using Lua scripting for atomicity:

-- atomicity-safe invalidation with stream update
local key = KEYS[1]
local stream = KEYS[2]
redis.call('DEL', key)
redis.call('XADD', stream, '*', 'action', 'invalidate', 'key', key)
return true

Another approach is using client-side caching (Redis 6+). This allows your application nodes to track which keys they have cached, and Redis will push a message to specific clients only when those keys change. This drastically reduces network overhead.
